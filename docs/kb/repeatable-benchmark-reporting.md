# Repeatable Benchmark Reporting

This document describes how to turn a set of poker session artifacts into a repeatable benchmark review like the ad hoc session review drafted for the `akg-recent-vs-stateless` runs.

The goal is not to claim AKG superiority from early runs. Current `llm-akg` should be reported as `llm-akg-recent`: a deliberately naive bounded-memory baseline that injects an opponent profile plus the last 5 completed hands. The report should make that framing explicit so later, richer AKG memory strategies can be compared against this baseline, and eventually against `llm-fullhistory` on both skill and cost.

## Why this report exists

Short heads-up NLHE sessions are noisy. A few large showdown pots, seat assignment, deal order, invalid-action fallbacks, and LLM nondeterminism can dominate a 200-hand result. A repeatable report should therefore separate:

1. **Raw session outcomes** — who won each run.
2. **Mirror-corrected outcomes** — what happened after each strategy played both seats on the same seed.
3. **Behavioral shape** — aggression, folds, showdown vs non-showdown chip flow.
4. **Efficiency** — token usage, cost, and cost growth when available.
5. **Evidence posture** — whether the run is a smoke test, baseline calibration, or thesis-bearing comparison.

For current `llm-akg-recent` vs `llm-stateless`, the report is primarily a baseline and behavior/cost characterization. It should not say that durable structured AKG memory has been proven better.

## Inputs

A report generator should accept either explicit session directories or a matchup/seed pattern.

Minimum required artifacts per session:

- `sessions/<id>/manifest.json`
- `sessions/<id>/hands.jsonl`
- `sessions/<id>/report.md` is optional; prefer deriving metrics from manifest + hands to avoid compounding prior report mistakes.

Optional artifacts for cost reporting:

- per-agent Pi usage logs, when present under session agent directories
- `pi-session.jsonl` or equivalent runtime usage files
- any future normalized usage summary generated by the LLM agent runtime

The first implementation can produce chip/behavior reports without token metrics, then fill efficiency sections only when usage artifacts are present.

## Pairing model

Use mirror pairs as the core unit of analysis.

A mirror pair is two sessions with:

- same seed
- same hand count
- same game settings
- same two strategies
- opposite seat assignment

Example:

```text
akg-recent-vs-stateless-seed2-a: seat0=llm-akg,       seat1=llm-stateless
akg-recent-vs-stateless-seed2-b: seat0=llm-stateless, seat1=llm-akg
```

The report should validate this pairing and warn if any of these differ:

- seed
- hand count
- variant
- blinds
- starting stack
- info realism
- model/provider, once that metadata is persisted in manifest
- strategy versions

## Required metrics

### Session-level table

For each session:

- session id / directory
- seed
- seat 0 strategy
- seat 1 strategy
- winner
- winner chip delta
- duration
- showdown count/rate
- preflop-only count/rate
- biggest pot or biggest chip swing
- invalid/fallback action counts (`auto_fold`, `auto_check`)

### Mirror-corrected strategy totals

Across all selected sessions:

- total chip delta per strategy
- chips/hand per strategy
- BB/100 per strategy, where BB is 2 chips in the current 1/2 game
- per-seed mirror aggregate for each strategy
- number of sessions won by each strategy
- number of mirror pairs won by each strategy

Do not treat individual session wins as the primary evidence when mirror pairs are available.

### Seat-bias check

Always report aggregate seat performance:

- total chip delta for seat 0 and seat 1
- chips/hand by seat
- whether one seat won every session

If seat effects dwarf strategy effects, call that out plainly. This is especially important in early baselines.

### Showdown vs non-showdown chip flow

For each strategy, split chip delta into:

- showdown chip delta
- non-showdown chip delta

This identifies whether a strategy is winning lots of small folds or winning/losing at large revealed-card pots.

### Behavioral action summary

For each strategy:

- calls
- checks
- bets
- raises
- folds
- aggressive actions = bets + raises
- fallback actions = `auto_fold` + `auto_check`

Useful derived rates:

- aggression share: `(bet + raise) / non-blind actions`
- fold share: `fold / non-blind actions`
- showdown rate by session

Avoid over-interpreting poker skill from these alone. They are descriptive diagnostics.

### Key hands

List the top N hands by absolute chip swing, ideally with:

- session
- hand number
- winner/loser strategy
- hero/villain hole cards by seat
- board
- terminal street
- chip delta
- whether showdown was reached

The report should use these as variance diagnostics, not cherry-picked proof.

### Cost and token metrics

When usage data exists, report these for each memory-bearing strategy and matchup:

- decisions
- prompt tokens
- completion tokens
- total tokens
- total cost
- average cost/hand
- average tokens/decision
- first-20-hand cost/hand
- last-20-hand cost/hand
- growth factor: `last_20_cost_per_hand / first_20_cost_per_hand`
- chips per 1k tokens or BB per 1M tokens

This is critical for the eventual `llm-akg-durable` vs `llm-fullhistory` comparison. `llm-fullhistory` may improve play by spending much more context; the benchmark must show both skill and cost.

## Report sections

A generated report should have this structure:

```markdown
# Benchmark Review: <matchup label>

## Framing

State what this comparison is and is not. For example:

- `llm-akg-recent` is a naive bounded-memory baseline using recent hands, not the durable AKG thesis agent.
- This run is baseline calibration / smoke testing / final comparison.
- Claims require mirror-corrected aggregates across enough seeds.

## Inputs

List included sessions and validation status.

## Headline

One short paragraph with the mirror-corrected result and the main caveat.

## Raw session results

Table of per-session winners.

## Mirror-corrected aggregate

Strategy totals, chips/hand, BB/100, pair wins.

## Seat-bias check

Seat 0 vs seat 1 totals and warning if seat effects dominate.

## Behavioral diagnostics

Showdown/non-showdown chip flow, action summaries, fallback actions.

## Key hands

Top swing hands.

## Cost and context efficiency

Only when usage data exists. Include token and cost growth.

## Interpretation

Conservative read of what the result means.

## Recommended next step

Suggested additional seeds, strategy changes, or report instrumentation.
```

## Interpretation rules

Use conservative language by default.

Recommended phrasing for early `llm-akg-recent` runs:

- “This is a shallow bounded-memory baseline, not evidence for durable AKG superiority.”
- “The result is dominated by seat/card-stream variance.”
- “The useful signal is behavioral and cost shape, not a thesis claim.”
- “AKG-recent won non-showdown chips but lost large showdowns,” if supported by the metrics.

Avoid:

- “AKG is better” from one or two mirror pairs.
- “Stateless is better” from a small aggregate with high showdown variance.
- Any claim that ignores seat-bias totals.
- Comparing `llm-akg-recent` to `llm-fullhistory` as if it were the final durable AKG strategy.

## Implementation plan

Add a small report command rather than hand-writing these reviews.

Suggested command shape:

```bash
./poker-report \
  -sessions sessions/akg-recent-vs-stateless-seed1-a,sessions/akg-recent-vs-stateless-seed1-b,sessions/akg-recent-vs-stateless-seed2-a,sessions/akg-recent-vs-stateless-seed2-b \
  -label akg-recent-vs-stateless \
  -out reports/akg-recent-vs-stateless.md
```

Possible later convenience mode:

```bash
./poker-report \
  -matchup akg-recent-vs-stateless \
  -seeds 1,2,3,4,5 \
  -sessions-root sessions \
  -out reports/akg-recent-vs-stateless.md
```

### Suggested packages

Keep the implementation small:

- `cmd/poker-report` for CLI parsing and file output
- `internal/reporting` for session loading, validation, aggregation, and Markdown rendering

### Data flow

1. Load each `manifest.json`.
2. Load each `hands.jsonl`.
3. Build a normalized in-memory model:
   - session metadata
   - seat-to-strategy map
   - per-hand result records
   - actions by strategy
4. Validate mirror groups by seed and strategy pair.
5. Compute session metrics.
6. Compute aggregate strategy, seat, showdown, non-showdown, action, and key-hand metrics.
7. Optionally attach usage/cost metrics if usage artifacts are found.
8. Render Markdown.

### Validation failures

The command should fail for malformed JSON or missing required artifacts.

It should warn, but still render, for:

- unpaired sessions
- mismatched session directory vs manifest `session_id`
- missing optional usage artifacts
- fallback actions present
- mirror pair metadata mismatch

## Future comparison ladder

Use this same report format across strategy generations:

1. `llm-akg-recent` vs `llm-stateless`
   - Purpose: low-cost shallow-memory baseline; behavior and cost characterization.
2. richer AKG candidate(s) vs `llm-akg-recent`
   - Purpose: prove new memory retrieval improves over the naive last-5-hands baseline.
3. best AKG candidate vs `llm-stateless`
   - Purpose: confirm improvement over no memory.
4. best efficient AKG candidate vs `llm-fullhistory`
   - Purpose: final skill/cost comparison against naive high-context memory.

The final report should compare both:

- **skill:** chips/hand, BB/100, mirror-pair wins, confidence intervals when enough hands exist
- **efficiency:** tokens/hand, cost/hand, context-growth slope, BB per token budget

## Example headline for the current baseline

> Across four 200-hand sessions, the naive `llm-akg-recent` baseline did not show a strategy advantage after mirror correction. Seat 0 won every session, indicating seat/card-stream variance dominated the raw outcomes. The useful signal is behavioral: `llm-akg-recent` was more aggressive and won more non-showdown chips, while stateless won more at large showdowns. Treat this as baseline characterization, not evidence for or against durable AKG memory.
